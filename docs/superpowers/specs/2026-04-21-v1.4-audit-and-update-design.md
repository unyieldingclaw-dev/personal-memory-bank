# Memory Bank Standard v1.4.0 — Audit & Update Design

**Date:** April 21, 2026
**Version target:** 1.4.0
**Scope:** Comprehensive correctness pass — bugs, inconsistencies, coverage gaps, missing features, 2025 best-practice additions, and token optimization

---

## Goal

Fix every confirmed bug and inconsistency found in the v1.3.1 audit, close coverage gaps between delivery vehicles (CLAUDE.md vs AGENTS.md vs .mdc files), add a Bash mb utility for Mac/Linux parity, incorporate the three 2025 best-practice rules that belong in always-loaded files, and trim enterprise-logging.mdc from 257 to ~65 lines — leaving the standard with better coverage and ~880 fewer tokens loaded per request than today.

---

## Architecture

No structural changes to the project layout. All changes are content edits to existing files, plus:
- One new file: `scripts/mb.sh` (Bash parity for mb.ps1)
- One new standard: `standards/SUPPLY-CHAIN.md`
- One new standard: `standards/MCP-SECURITY.md`
- One new doc: `docs/TELEMETRY-GUIDE.md`
- One new directory: `docs/superpowers/specs/` (this file)

The always-loaded file set (CLAUDE.md, AGENTS.md, 5× .mdc) grows by 7 lines net before the logging trim, then shrinks by ~190 lines after the trim. Net: −14% token cost vs. today.

## Bucket 1: Bugs (wrong content)

### B1 — `standards/LOGGING.md`: `logger.timing()` does not exist

`logger.timing("operation_name")` is shown as a structlog quick-reference pattern. structlog has no `.timing()` context manager. Replace with the correct manual timing pattern:

```python
start = time.time()
result = operation()
logger.info("operation_completed", duration_ms=(time.time() - start) * 1000)
```

### B2 — `standards/LOGGING.md`: `logger.isEnabledFor()` is stdlib, not structlog

`logger.isEnabledFor(logging.DEBUG)` is a method on the stdlib `logging.Logger`, not on a structlog `BoundLogger`. Remove the line or replace with a note that structlog defers level filtering to the underlying stdlib handler.

### B3 — `standards/LOGGING.md`: Dead references to non-existent extension files

Lines reference `extensions/logging-typescript.md` and `extensions/logging-go.md`. Neither exists. Replace with: "TypeScript logging extension: see `standards/extensions/typescript.md` for pino patterns. Go and Java extensions are planned (see progress.md)."

### B4 — `standards/LOGGING.md`: Dead reference to `examples/logging/`

The examples/ directory is empty and not planned in progress.md. Remove the reference. Point readers to the logging extension files and `docs/LOGGING-GUIDE.md` for runnable examples.

### B5 — `templates/AGENTS.md`: `DELETE without WHERE` severity mismatch

AGENTS.md lists `DROP TABLE without WHERE` in the BLOCK tier. Every other file (`CLAUDE.md`, `security.mdc`, `standards/SECURITY-GUARDRAILS.md`) places it in CONFIRM. CONFIRM is correct — the operation is dangerous but not unconditionally refused (legitimate admin operations exist). Fix AGENTS.md to match CONFIRM.

## Bucket 2: Inconsistencies (conflicting content between files)

### I1 — `templates/CLAUDE.md` BLOCK tier is shorter than `security.mdc`

security.mdc BLOCK includes `git reset --hard on shared branches without explicit backup` and `modify git config`. CLAUDE.md BLOCK does not. Add both to CLAUDE.md BLOCK.

### I2 — `templates/CLAUDE.md` CONFIRM tier is shorter than `security.mdc`

security.mdc CONFIRM includes: bulk operations on >3 files, `git push --force` to any branch (not just main), interactive rebase, schema changes to production, CI/CD file modification. CLAUDE.md CONFIRM has only 5 items and misses all of these. Sync CLAUDE.md CONFIRM to match security.mdc.

### I3 — `templates/cursor/rules/workflow.mdc` skip threshold not in canonical standard

workflow.mdc says "change is obvious and < 20 lines → start at Phase 4." `standards/WORKFLOW.md` and `templates/AGENTS.md` say "single-file fix, typo, config value change" — no line count. The line-count heuristic is useful and concrete. Add it to both `standards/WORKFLOW.md` and `templates/cursor/rules/workflow.mdc` so the two files match.

### I4 — Archive destination is inconsistent across files

`standards/MEMORY-BANK.md` quick commands say archive moves history "to AGENTS.md." `mb.ps1` Show-Archive says the same. But MEMORY-BANK.md body says old content can go to "docs/". Standardize: old session history goes to `docs/ARCHIVE.md` (a file teams create if needed), not AGENTS.md. AGENTS.md is a live rules file, not a history store. Update all references.

### I5 — `templates/claude-commands/feature-dev.md` allowed-tools list is misleading

The allowed-tools frontmatter permits only git commands. Phases 3–5 of the workflow require reading files, writing files, running tests, and running linters — none of which are in the list. Claude Code's actual tool permissions are broader than slash command declarations, so this doesn't break anything, but it misleads developers reading the file. Either expand the list to reflect reality or remove the allowed-tools restriction entirely and add a comment explaining why.

## Bucket 3: Coverage gaps (standards present but missing from some delivery vehicles)

### C1 — `templates/CLAUDE.md` is missing the Workflow standard

CLAUDE.md has Memory Bank, Security, Code Quality, Logging, and Handoff — but no 7-phase workflow. Every other delivery vehicle has it. Add a Workflow section to CLAUDE.md matching the lean version in AGENTS.md (7 bullet points, skip condition, ~12 lines).

### C2 — `templates/AGENTS.md` is missing the Logging standard

AGENTS.md has Memory Bank, Security, Workflow, Code Quality, and Token Discipline — but no Logging. Any tool reading only AGENTS.md (Codex, Gemini CLI) produces unstructured logs. Add a Logging section to AGENTS.md: the core rule, the good/bad example, and the four anti-patterns (~15 lines, no code block examples).

### C3 — IP address in FastAPI logging example is unwarned PII

`standards/extensions/logging-python.md` logs `client_ip=request.client.host` in the FastAPI middleware example. IP addresses of natural persons are PII under GDPR. The PIISanitizer processor does not redact IPs by default. Add a comment to that example: "Note: IP addresses are PII under GDPR. Log only if your privacy policy covers it and your log access controls restrict it."

### C4 — `memory-bank/systemPatterns.md` still says "Three Standards Architecture"

The project has had five standards since v1.2.0/v1.3.0. Update systemPatterns.md to reflect the current five-standard architecture.

### C5 — `memory-bank/techContext.md` has a stale file tree and wrong global rules claim

The file tree omits all v1.2.0 and v1.3.0 additions (LOGGING.md, WORKFLOW.md, AGENTS.md, claude-commands/, enterprise-logging.mdc, workflow.mdc, CLAUDE-CODE-PLUGINS.md). It also states "No user-level rules supported — each project needs its own CLAUDE.md" — explicitly corrected everywhere else in v1.3.1 but missed here. Update both.

## Bucket 4: Missing features

### F1 — No `mb.sh` (Bash equivalent of mb.ps1)

All documentation describes `mb status`, `mb update`, `mb archive`, `mb slim`, and `mb commit` as if they are cross-platform. They are Windows-only. Mac/Linux teams have no utility.

Create `scripts/mb.sh` implementing all five commands with equivalent behavior. Follow the same patterns as `init-memory-bank.sh`: `set -e`, ANSI color variables, `POSIX`-safe constructs. Use `wc -l` for line counts (POSIX equivalent of `Measure-Object -Line`). The commit command uses `git add memory-bank/` and `git commit -m "chore: Update Memory Bank context"` — same as the PS1 version.

Update both init scripts to reference `mb.sh` in the summary output for Unix targets.

### F2 — Training exercises are Windows-only

All four exercises use PowerShell commands (`Get-ChildItem`, `Get-Content`, `Test-Path`). Add a Bash equivalent command after each PowerShell example in all four exercise files, formatted as a collapsible or labeled "Mac/Linux:".

## Bucket 5: 2025 Best Practices

### N1 — Supply chain / slopsquatting (7 lines added to always-loaded files)

~20% of AI-generated code references packages that do not exist. Attackers can reliably predict which hallucinated packages will repeat and register them. No mitigation exists anywhere in the standard.

**In `security.mdc` BLOCK section (3 lines):**
```
- **NEVER** recommend installing a package without first verifying it exists on
  the official registry (PyPI, npm, etc.) — hallucinated package names are a
  live supply-chain attack vector (slopsquatting)
```

**In `CLAUDE.md` BLOCK section (2 lines):**
```
- Recommending unverified packages (verify existence on official registry before suggesting install)
```

**In `AGENTS.md` BLOCK section (2 lines):**
```
- Recommending packages without verifying they exist on the official registry (slopsquatting risk)
```

Full rationale, mitigation steps, and SCA tool recommendations go in `standards/SUPPLY-CHAIN.md`.

### N2 — Rules file integrity (0 lines in always-loaded files)

The Unicode injection attack embeds hidden control characters in `.cursor/rules/*.mdc` or `CLAUDE.md` to inject silent instructions. An always-loaded rule cannot protect against this — the compromised file loads before any integrity check in it. This is a human/CI review concern only.

No changes to always-loaded files. Add guidance to `standards/SUPPLY-CHAIN.md`: recommended pre-commit hook to reject non-ASCII characters in `.mdc` file headers, and a developer checklist for reviewing community or vendor rule files before adoption.

### N3 — Context file exclusion list (2 lines in always-loaded files)

No file currently states what must not go in memory-bank/ files. The AI writes these files on every session; without a rule it may cache secrets or production data into persistent context.

**Add to Memory Bank section of both `CLAUDE.md` and `AGENTS.md` (2 lines each):**
```
- Never write secrets, credentials, API keys, PII, production data samples,
  or full code dumps to memory-bank/ files
```

Full exclusion rationale goes in `standards/MEMORY-BANK.md`.

### N4 — MCP credential security (2 lines in always-loaded files)

MCP config files (`mcp.json`, `.mcp.json`) frequently contain hardcoded API keys and tokens. The AI is often the one generating or editing these files.

**Add to `security.mdc` and `CLAUDE.md` BLOCK sections (2 lines each):**
```
- **NEVER** hardcode credentials, tokens, or API keys in `mcp.json` or any
  MCP configuration file — use environment variable references instead
```

Full MCP security guidance (tool poisoning, server provenance, least-privilege per tool) goes in `standards/MCP-SECURITY.md`.

### N5–N7 (0 lines in always-loaded files)

Telemetry/data residency (N5), hard vs. soft enforcement mapping (N6), and developer confidence gap (N7) are human configuration, DevOps documentation, and training material respectively. None belong in always-loaded files.

- N5 → `docs/TELEMETRY-GUIDE.md`: per-tool telemetry opt-out settings
- N6 → new section in `standards/SECURITY-GUARDRAILS.md`: enforcement level table
- N7 → `training/` note in presentation and exercises

## Token Optimization: Trim `enterprise-logging.mdc`

The file is 257 lines / ~1,562 tokens. 56% is code block examples. The OpenSSF guidance is that rule files over ~150 lines show compliance degradation.

**Keep in enterprise-logging.mdc (~65 lines):**
- Core rule statement (structured logging always)
- One good/bad comparison (the short 6-line version)
- Error logging pattern (the most distinct pattern not obvious from the rule)
- Log levels table
- Four anti-pattern bullet points (no code blocks — prose only)
- Pointer to `standards/LOGGING.md` for full examples

**Move to `standards/LOGGING.md`** (already exists, has room):
- Business Events full examples
- External API Calls block
- Full Correlation IDs block
- All four full bad/good code block pairs
- PII table
- Environment Variables table
- Python/TypeScript boilerplate implementation
- "Before Claiming Done" checklist

Result: enterprise-logging.mdc shrinks from ~1,562 tokens to ~550 tokens, saving ~1,000 tokens per Cursor request.
